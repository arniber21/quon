# Neutral-atom backend: architecture model

This document is the normative reference for the `quon_na` neutral-atom backend
(ADR-0007, ADR-0008, ADR-0009). It defines the hardware model the backend
approximates, the vocabulary and constraints every pass is built against, every
field of the neutral-atom target JSON schema (issue #101), and the
per-`CodeFamily` overhead formulas (issue #109). Every modeled mechanism is
attributed to the specific paper it comes from, and every numeric constant is
either pinned to a cited source or explicitly labeled an illustrative
placeholder.

Citation keys like `[OLSQ-DPQA]` resolve in the References section at the end.
Companion document: [literature_notes.md](./literature_notes.md) summarizes each
source and what this backend takes from it.

Vocabulary follows `CONTEXT.md` ("Neutral-atom backend" section): *atom*,
*logical qubit*, *code block*, *AOD movement*.

## 1. What is being modeled

The backend targets an abstract **reconfigurable neutral-atom array**: atoms
held in optical traps on a 2D plane, where some traps are static (generated by
a spatial light modulator, SLM) and some are mobile (generated by a crossed 2D
acousto-optic deflector, AOD). Entangling gates are mediated by the Rydberg
blockade between atoms brought within interaction range; connectivity is
reconfigured at runtime by physically moving atoms rather than by inserting
SWAP gates. This is the architecture family called DPQA (dynamically
field-programmable qubit arrays) in [OLSQ-DPQA] and [Enola], and FPQA/RAA in
[Atomique].

Two variants are modeled, in two stages (issues #106 and #107):

1. **Flat reconfigurable array** — a single plane, one SLM grid plus one or
   more AOD grids, a global Rydberg laser illuminating the whole plane. "Which
   pairs may interact" is a *distance and scheduling* notion, not a spatial
   one. This is the model of [OLSQ-DPQA], [Enola], and [Atomique].
2. **Zoned architecture** — physically separate regions with different
   capabilities: a *storage zone* (dense static traps, shielded, long
   coherence), an *entanglement zone* (paired traps under a zone-restricted
   Rydberg beam), and optionally a *readout zone* (mid-circuit measurement
   without disturbing other atoms). This is the model of [AbstractModel],
   [RAP], and the [QMAP-docs] zoned compiler, following the experimental
   architecture of [Bluvstein24].

These are **two genuinely different hardware models from two different lines of
literature**, and the backend builds both deliberately: the flat AOD movement
planner first (#106), the zoned joint placement-routing scheduler second
(#107). Section 4 attributes each mechanism to its source so the combination is
never mistaken for a single reproduced architecture.

## 2. What is explicitly not modeled

- **No full QEC decoder.** The QEC layer (issue #109) is a logical-op and
  resource-accounting abstraction: code blocks expand to atom counts by the
  formulas in Section 10, and syndrome rounds are schedulable operations, but
  no syndrome decoding, error propagation, or logical failure-rate simulation
  is performed.
- **No vendor-specific hardware claims.** The model is grounded entirely in
  published techniques and their stated parameters. The target descriptor is a
  *generic* reconfigurable neutral-atom machine; it does not describe, and must
  not be read as describing, any specific commercial device.
- **No pulse-level physics.** Gates are discrete scheduled actions with
  durations and fidelities; no Hamiltonian, pulse-shape, or blockade-strength
  simulation.
- **No atom-loss or heating simulation.** [Atomique] models movement-induced
  heating and probabilistic atom loss (its Eqs. 1–2); this backend's v0 cost
  model does not. Movement cost is time-based (Section 9). Heating-aware cost
  is a possible later extension and would cite [Atomique] Sec. IV.
- **No continuous trajectories.** Movement is modeled at the granularity of
  rearrangement steps between discrete placements, as in [RAP] Sec. III-A;
  collision-freedom within a step is guaranteed by the AOD ordering
  constraints (Section 5), not by path simulation.

## 3. Fixed, reconfigurable, and zoned arrays

Three architecture families appear in the neutral-atom compilation literature.
The distinction matters because compiler problems and cost models differ per
family:

| Family | Traps | Connectivity | Compiler problem |
| --- | --- | --- | --- |
| **Fixed array** | Static only | Fixed local neighborhood (e.g. within an interaction radius) | Qubit mapping + SWAP insertion, as on any fixed-coupling device. Baselines "FAA" in [Atomique] Sec. V-A. |
| **Reconfigurable array (DPQA/FPQA)** | Static SLM + mobile AOD | Any pair can be brought within range by movement | Joint placement, movement scheduling, gate layering under AOD constraints ([OLSQ-DPQA], [Enola], [Atomique]) |
| **Zoned architecture** | Static traps in operation-specific zones + AOD transport between zones | Entangling only inside the entanglement zone; storage shielded; readout isolated | Zone-aware placement and routing; shuttle scheduling storage↔entanglement↔readout ([AbstractModel], [RAP], [Bluvstein24]) |

Quon's existing gate-model backend (`TargetKind::Fixed`) already covers the
first family generically. The `quon_na` backend covers the second and third via
`TargetKind::NeutralAtomReconfigurable` (ADR-0009).

## 5. The movement model: AOD row/column coupling

**This backend does not model free-grid Manhattan movement.** Atoms in an AOD
are trapped at the intersections of a set of AOD rows and columns; the only
movement controls are the Y coordinate of each row and the X coordinate of each
column ([OLSQ-DPQA] App. A.2, Implication 5: "If the array is the AOD type, a
row/column of traps move together"; Sec. 2: "we cannot move AOD traps
individually"). A movement plan that moves one atom independently of its row
and column mates is not realizable on this hardware model and must be rejected
by the `quantum.na` movement-legality verifier (#102, #106).

The movement constraints, as enforced by the backend:

- **M1 — Coupled motion.** All atoms in an activated AOD row (column) move
  together when that row (column) moves. Source: [OLSQ-DPQA] Sec. 2 and App.
  A.2 Implication 5; identically [Atomique] Sec. II.
- **M2 — Order preservation (no crossing).** Within one AOD, a row cannot move
  past another row, and a column cannot move past another column; the
  row/column order at the start of a move is the order at the end. Source:
  [OLSQ-DPQA] App. A.2 Implication 6 ("a row cannot cross over another row, a
  column cannot move over another column"); [Atomique] Sec. III-C Constraint 2;
  [AbstractModel] Sec. III-C constraint (a). Physical reason: crossing rows
  would violate the minimum row separation and cause heating/atom loss
  ([OLSQ-DPQA] Sec. 2).
- **M3 — No merging.** Two rows (columns) of the same AOD cannot occupy the
  same coordinate. Source: [Atomique] Sec. III-C Constraint 3; equivalently
  [Enola] Sec. 5 conflict types 1–2 (same source row must share a destination
  row and vice versa); [OLSQ-DPQA] App. A.1 Implication 3 (minimum intra-array
  row/column separation d_s).
- **M4 — Static traps are static.** Atoms in SLM traps do not move; position
  changes for an SLM-trapped atom require a trap transfer into an AOD (a
  costed, fidelity-bearing action). Source: [OLSQ-DPQA] App. A.2 Implication 4,
  App. A.5.
- **M5 — Occupancy.** One trap holds at most one atom; two atoms may never
  occupy the same site at the same time (collision detection in #106). Source:
  [OLSQ-DPQA] App. A.1 Implication 1.

Constraints M2 and M3 bind **within one AOD only**; rows/columns of different
AODs may cross ([Atomique] Sec. II; [OLSQ-DPQA] Sec. 4 duplicates the ordering
constraint per AOD).

In the zoned model the same constraints apply, restated as the three
rearrangement constraints of [RAP] Sec. II-A Fig. 2 (after [AbstractModel]
Sec. III-C): *non-crossing*, *preservation* (atoms starting in the same AOD row
stay in it), and *ghost spots* (activating rows and columns affects **all**
their grid intersections, so loading a subset of atoms needs offset moves —
this is why zone transfer is not free even when distances are short).

**Movement timing.** One rearrangement step covering maximum distance d takes

```
t_move(d) = sqrt(d / a),  a = 2750 m/s²   (e.g. d = 110 µm → t = 200 µs)
```

cited to [RAP] Sec. VI-B and [Enola] Sec. 2; [OLSQ-DPQA] App. A.2 states the
same law as t = T0·√(D/D0) with T0 = 200 µs, D0 = 110 µm ("to maintain constant
heating"). Each trap transfer (pick-up or drop-off) adds 15 µs ([RAP] Sec.
VI-B; [Enola] Sec. 2). **Known literature divergence:** [Atomique] instead
charges a fixed 300 µs per movement stage regardless of distance (its Table I)
and lets distance enter only its heating model. The backend implements the
√-law because it is what the reproduced paper's metric uses ([RAP] Sec. VI-B);
the divergence is documented here so nobody "fixes" one to match the other.

## 6. The interaction model: Rydberg range and isolation

For the flat array ([OLSQ-DPQA] App. A.3):

- **R1 — Range.** Two atoms can perform an entangling gate iff their distance
  is ≤ r_b (the Rydberg blockade radius) while the Rydberg laser is on
  (Implication 8).
- **R2 — Compulsion.** The flat-array Rydberg laser is global: every pair
  within r_b when the laser fires undergoes a gate, wanted or not (Implication
  9). A movement plan that parks two non-partner atoms within r_b at a Rydberg
  stage is illegal ([Atomique] Sec. III-C Constraint 1 rejects such gate sets).
- **R3 — Isolation.** Non-interacting atoms must be separated by > 2.5 × r_b;
  among any three atoms at most one pairwise distance may be < 2.5 × r_b when
  the laser is on (Implication 10). This is the source of the schema field
  `min_rydberg_spacing_um` and of the interaction-site pitch used by the flat
  planner.
- **R4 — Stage-based error accounting.** Because the laser is global, every
  illuminated idle atom accrues error each Rydberg stage ([Enola] Sec. 2 models
  this as f_exc per idle atom-stage). Minimizing the number of Rydberg stages
  is therefore the correct flat-array objective, not minimizing gate count
  ([Enola] Secs. 2, 7).

In the zoned model, R1–R3 hold *inside the entanglement zone*: traps there come
in pairs placed so that pair members interact and distinct pairs do not
([RAP] Sec. II-A); the Rydberg beam covers only that zone, so storage-zone
atoms are shielded and R4's idle penalty applies only to atoms left isolated in
the entanglement zone ([AbstractModel] Sec. III-A; [RAP] Sec. II-A).

## 7. The zoned model

Zone taxonomy, per [AbstractModel] Sec. III-A (formalizing [Bluvstein24] Fig.
1a):

- **Storage zone** — densely packed static traps; shielded from the Rydberg
  beam; long coherence; 1Q rotations allowed.
- **Entanglement zone** — spatially separated *pairs* of static traps under the
  zone's Rydberg beam; all pairs execute CZ in parallel when the beam fires;
  a lone atom in the beam performs a noisy identity.
- **Readout zone** — measures every atom inside it simultaneously without
  disturbing atoms elsewhere; enables mid-circuit measurement. (Modeled by
  [AbstractModel]; [RAP] models only storage + entanglement. The schema keeps
  readout zones optional accordingly.)

Executing a 2Q gate means: load the atom from a static trap into the AOD
(15 µs), move it (√-law), drop it into an entanglement-zone trap, fire the
beam, and move it back — the load/move/store step sequence of [AbstractModel]
Sec. III-B. The reuse optimization ([RAP] Sec. III-A) keeps an atom in the
entanglement zone when its next 2Q layer uses it again, saving two transfers.

The zoned scheduler (#107) reproduces the **joint placement-routing
formulation** of [RAP]: placements are chosen per 2Q layer by an A* search
(Sec. IV-B) whose node cost is a *routing* cost — implied movements are grouped
greedily into AOD-compatible parallel groups, and cost(p) = Σ_G √(d_max(G))
over groups G (Eq. (1)), extended with reuse and one-layer look-ahead terms
(Eq. (2)) and guided by the heuristic of Eqs. (3)–(5). Placement quality is
thus measured in rearrangement steps and durations, not raw travel distance —
that is what "routing-aware"/joint means, versus the sequential
distance-minimizing placement of the ZAC baseline it improves on ([RAP] Secs.
III-B, VI-B). Issue #107 carries the reproduction requirements; issue #111
pins the regression anchor to [RAP] Table I (e.g. the 42-qubit `ising`
benchmark: 22 rearrangement steps routing-agnostic vs 9 routing-aware).

## 8. Target JSON schema

The `NeutralAtomTarget` payload of `TargetKind::NeutralAtomReconfigurable`
(ADR-0009, issue #101). `quonc --target <path>` loads it like any other target;
the loader dispatches on `kind`. Field names are normative for #101; the
checked-in sample is `targets/neutral_atom/generic_rna_v0.json` with
`id = "generic_reconfigurable_neutral_atom_v0"`.

Units: lengths in µm, times in µs, fidelities as probabilities in [0, 1].

### 8.1 Top level

| Field | Type | Meaning | Compiler constraint it feeds |
| --- | --- | --- | --- |
| `id` | string | Target identifier (shared across all `TargetKind`s) | Report labeling |
| `kind` | string, `"neutral_atom_reconfigurable"` | `TargetKind` discriminant | Loader dispatch |
| `grid` | object | Overall bounding box: `width_um`, `height_um` | All placements and moves must stay in bounds |
| `zones` | array of Zone | Zone list (§8.2) | Zone occupancy/capacity/operation legality (#107) |
| `movement` | object | AOD movement model (§8.3) | Movement legality verifier, move-time cost (#102, #106) |
| `interaction` | object | Rydberg parameters (§8.4) | Entangling legality, isolation spacing (#105, #106) |
| `native_gates` | array of string | Gate names executable natively, e.g. `["cz", "rz_local", "ry_global", "measure_z"]` | Lowering must decompose to this set |
| `timing` | object | Operation durations (§8.5) | Schedule makespan, decoherence-weighted cost |
| `fidelity` | object | Operation fidelities + coherence time (§8.5) | Resource report fidelity estimate (#110) |
| `cost_model` | object | Linear cost weights (§9) | Scheduler objective, resource report |

### 8.2 Zone

| Field | Type | Meaning | Compiler constraint it feeds |
| --- | --- | --- | --- |
| `zone_id` | integer | Unique zone identifier | Cross-references in schedule actions |
| `kind` | `"storage"` \| `"entanglement"` \| `"readout"` | Zone capability (§7) | 2Q ops only in `entanglement`; measurement only in `readout` when `require_readout_zone` is set |
| `rows`, `cols` | integer | Static-trap grid extent (for `entanglement`: trap *pairs*) | Zone capacity = rows × cols; occupancy checks (#107) |
| `origin_um` | [number, number] | Lower-left corner of the zone's trap grid | Absolute site coordinates; movement distances |
| `site_pitch_um` | [number, number] | Trap spacing in x and y | Site coordinates; isolation checking |
| `pair_gap_um` | number, `entanglement` only | Distance between the two traps of a pair | Pair members must sit within r_b; distinct pairs beyond 2.5 r_b (§6) |

Zone shape follows the [QMAP-docs] architecture JSON (`storage_zones` /
`entanglement_zones` with SLM grids; entanglement zones as two interleaved
grids forming pairs), flattened into one list for `deny_unknown_fields`
loading. Validation (per #101): non-negative extents, no overlapping zones,
zones within `grid`.

A flat-array target (stage 1, #106) is expressed as a single `entanglement`
zone covering the whole grid: every site is then a legal interaction site and
the zone constraints degenerate to the flat model of [OLSQ-DPQA]/[Enola]. A
zoned target (stage 2, #107) declares at least one `storage` and one
`entanglement` zone.

## 9. Cost model

The v0 cost model is a simple linear functional over a compiled schedule,
reported by the resource estimator (#110) and minimized greedily by the
schedulers:

```
cost(schedule) = w_stage · n_rydberg_stages
              + w_move  · Σ_steps t_move(d_max(step))
              + w_xfer  · n_trap_transfers
              + w_idle  · Σ_atoms t_idle(atom)
```

Rationale per term, each grounded separately:

- `n_rydberg_stages`: the flat-array objective; every stage exposes all
  illuminated atoms to Rydberg error ([Enola] Secs. 2–3, R4).
- `Σ √(d_max)` per movement group: the [RAP] Eq. (1) placement cost — duration
  of a rearrangement step is set by its longest move under the √-law.
- Transfer count: each transfer is a fidelity-bearing action ([Enola] Sec. 2:
  99.9% per transfer, 4 per gate; [RAP] reuse analysis exists precisely to
  save transfers).
- Idle time: linear decoherence proxy, 1 − t/T ([Enola] Eq. (1) decoherence
  factor).

The **weights are illustrative placeholders** (Section 8.6): the literature
optimizes these terms directly rather than a weighted sum, so any particular
weighting is ours. The shape (which terms exist) is cited; the weights are not.
Snapshot tests may pin them for regression purposes but must not present them
as published values.

## 10. Code families and overhead formulas

The QEC abstraction layer (#109) expands each logical qubit's code block into
atoms by an exact per-`CodeFamily` formula. N below counts **atoms per logical
qubit including syndrome/check ancillas** unless stated otherwise. These
formulas are normative for #109 ("implement exactly what's documented there").

**Implementation:** `quon_na::qec` (`quon_na/src/qec.rs`) — `CodeFamily`,
`CodeBlock`, `LogicalOp`, `atoms_per_logical`, and `expand_code_block`. Toy
demo: `quon_na/examples/repetition_code_toy.rs`. No source-language syntax;
frontend linearity is unaffected.

### 10.1 `SurfaceCodeLike`

```
N(d) = 2d² − 1        (d² data atoms + (d² − 1) check atoms), d odd, d ≥ 3
```

Source: the rotated surface code. [Bravyi24] Sec. 1 states it directly: "the
standard surface code architecture encodes k = 1 logical qubit into n = d² data
qubits for a distance-d code and uses c = n − 1 check qubits"; the d²-data-qubit
([[d², 1, d]]) construction originates with [BMD07]. The common shorthand
"N ≈ 2d²" (used in issue #109) is this formula rounded; the implementation uses
the exact 2d² − 1.

**Citation trap, recorded deliberately:** [Fowler12] — the canonical surface-
code reference — does *not* give 2d²; its unrotated planar patch is
n_q = (2d − 1)² (its Sec. VII, Fig. 6). [Fowler12] remains the citation for the
logical error-rate scaling p_L ≅ 0.03 (p/p_th)^((d+1)/2) with p_th ≈ 0.57% per
step (its Eqs. (10)–(11), Fig. 4), should the resource estimator later report
it.

### 10.2 `RepetitionCodeToy`

```
N(d) = 2d − 1         (d data atoms + (d − 1) measure atoms), d ≥ 2
```

Source: [Kelly15], the canonical repetition-code experiment — "five data qubits
and four measure qubits in an alternating pattern" (its Fig. 1b; d = 5
instance, with the d = 3 five-qubit instance in its abstract). Toy/demo family
only (issue #109 ships a demo circuit); corrects bit-flips only, not a real
quantum memory.

### 10.3 `HighRateQldpcLike`

```
N(r) = ceil(1/r)      where r = k/n is the code rate (data atoms per logical)
```

Physical-per-logical overhead is the **inverse of the rate** — n/k = 1/r, not
k/r (the inversion is called out in issue #109 because it is an easy bug).
Source: [Bravyi24] Sec. 1 defines the net rate r = k/(n + c) over data plus
check qubits; for its bivariate-bicycle family c = n, so a target that accounts
check atoms should double the data-only figure (its Table 1 caption: "A code
with parameters [[n, k, d]] requires 2n physical qubits in total"). Sanity
anchor: [[144,12,12]] → 144/12 = 12 data-only, 288/12 = 24 with checks
([Bravyi24] Table 1). The `quon_na` implementation takes r as the *net* rate
(checks included), so N(r) already covers ancillas; a `[[144,12,12]]`-like
target sets r = 1/24. Constant-rate qLDPC families exist asymptotically
([PK22]), and [Xu24] Table I gives concrete overhead points on reconfigurable
atom arrays (e.g. 400 logical / 19 600 physical, a 6.9× saving over surface
code at p = 10⁻³) — the reason this family is worth modeling on this backend
at all.

### 10.4 `AbstractBlockCode`

```
N = ceil(n/k)         for a user-specified [[n, k, d]] block code
```

A flat, user-specified ratio with no built-in ancilla accounting — the escape
hatch for codes the other three families don't fit. The [[n,k,d]] notation:
[Gottesman97] §2.3 ("A distance d code encoding k qubits in n qubits is
described as an [n,k,d] code… often written… as [[n,k,d]]"). Whether n includes
check atoms is the user's choice; the target JSON must say which convention its
ratio uses.

### Summary

| `CodeFamily` | Formula (atoms per logical qubit) | Parameters | Primary citation |
| --- | --- | --- | --- |
| `SurfaceCodeLike` | 2d² − 1 (incl. checks) | distance d | [Bravyi24] §1; [BMD07] |
| `RepetitionCodeToy` | 2d − 1 (incl. measure atoms) | distance d | [Kelly15] Fig. 1b |
| `HighRateQldpcLike` | ceil(1/r) (r = net rate, incl. checks) | rate r | [Bravyi24] §1, Table 1; [PK22]; [Xu24] |
| `AbstractBlockCode` | ceil(n/k) (user convention) | n, k | [Gottesman97] §2.3 |

Code blocks are scheduling units: whole blocks move between zones and logical
2Q gates are physical-parallel transversal interleavings — the operational
picture motivating this layer, demonstrated in [Bluvstein24] (Figs. 1a, 2a).
That paper is motivation, not a reproduction target; no transversal-gate
scheduler is claimed for v0.
